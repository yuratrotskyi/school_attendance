# Дизайн: збір відвідуваності з `nz.ua/journal/list` (пагінація журналів)

## 1. Ціль
Перевести джерело даних з експорту звітів на журнали `https://nz.ua/journal/list`, щоб щоденна звітність формувалась із фактичних позначок відвідуваності в журналах, доступних акаунту.

## 2. Підхід (погоджено)
Варіант 2: `API-first + DOM fallback`.
- Спочатку пробуємо забрати структуру відвідуваності з мережевих JSON-відповідей журналу.
- Якщо структура не знайдена або невалідна, переходимо на DOM-збір за селекторами.
- Обидва шляхи нормалізують дані в один формат записів уроків.

## 3. Джерело і охоплення
- Точка входу: `https://nz.ua/journal/list`.
- Обробляємо всі журнали, доступні поточному користувачу.
- Для кожного журналу проходимо всі сторінки пагінації (очікувано до 9, але без hard-limit у коді).

## 4. Правила мапінгу відміток
- `Н` => `ABSENT` (враховується у відсутностях і в інцидентах).
- `ХВ` => ігнорується для метрик відсутності (не рахується як відсутність).
- Порожня клітинка або інші не-`Н` позначки => `PRESENT` (для відновлення послідовності уроків і правила "втік").

## 5. Правило інциденту "втік з уроку"
Без змін: інцидент фіксується, якщо після першої присутності за день є `1+` неповажний пропуск (`Н`).

## 6. Нормалізований формат
Кожна клітинка відвідуваності після мапінгу перетворюється на рядок:
- `student_id`
- `student_name`
- `class_name`
- `date`
- `lesson_no`
- `status` (`PRESENT`/`ABSENT`)
- `reason_code` (порожньо для `Н`, порожньо або службове значення для інших)

Для уникнення дублів (однаковий урок, підтягнутий з кількох джерел/сторінок) використовуємо ключ:
`journal_id + student_id + date + lesson_no`.

## 7. Надійність і контроль
- Якщо журнал не вдалось прочитати, логуємо помилку і продовжуємо з наступним журналом.
- Якщо після обходу не отримано жодного запису, повертаємо явну помилку пайплайна.
- У dry-run режимі можна передати локальний CSV як і раніше.

## 8. Зміни в конфігурації
Розширюємо `config/nz_selectors.example.json` секціями:
- `journal_list` (селектори списку журналів і пагінації),
- `journal_page` (селектори таблиці/клітинок або параметри для fallback),
- опційно `api_hints` (патерни URL/ключів для JSON).

## 9. Тестова стратегія
- Unit: пагінація, дедуплікація, мапінг `Н/ХВ`, побудова row-ключа.
- Unit: парсинг API payload у нормалізовані записи.
- Unit: DOM fallback parser на фікстурному payload.
- Integration (dry-run): пайплайн формує артефакти з локального CSV без збору.
